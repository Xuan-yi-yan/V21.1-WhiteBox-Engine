  双核辩论系统 (ToolAgent) — 架构与战绩
  开发时间: ~10分钟 | 代码量: ~400行 | 实际作用: 极大
═══════════════════════════════════════════════════════════════

一、设计理念

  两个AI互相审计, 比单独用一个AI更不容易出错。
  DeepSeek负责分析, 千问负责审查——一个出方案, 一个找漏洞。
  用户只做决策, 不做中间人。

二、技术架构

  ToolAgent类:
    - 每个Agent独立管理消息历史
    - 自动检测工具调用(JSON+XML双格式)
    - 工具去重(同一参数不重复调)
    - 文件缓存(30条历史, 避免重复读取)
    - 永久记忆(辩论结果持久化)

  7个工具:
    read_file / search_files / search_content / list_dir
    web_fetch / run_cmd / write_file

  辩论流程:
    DeepSeek第1轮 → 千问审查 → DeepSeek改进 → 千问二审 → 收敛判断
    每轮支持用户打断注入新想法

  关键特性:
    - DeepSeek只读模式(不能写文件/执行命令)
    - 千问完整权限(可读代码验证)
    - max_tokens=32768 (防截断)
    - 轮流标记"← XX完毕" (防抢答)
    - 收敛判断: 千问回复CONVERGE即结束

  鲁棒性:
    - 千问失败自动重试(等5秒)
    - 参数别名映射(千问XML格式兼容)
    - 重复工具调用自动跳过

三、辩论记录 (V21开发期间, 全部9轮)

  第1轮: gate叠加问题
    发现: gate_input加法让prev_loss淹没ABC
    修复: 调整初始化尺度

  第2轮: ABC颗粒度对齐
    争议: 64D→128D还是256D?
    结论: 128D最优, 3层126槽

  第3轮: 爆炸根因 (第1次)
    假说: gate饱和/aux阶跃/explore无tanh
    验证推翻: 三个假说全部不成立

  第4轮: 爆炸根因 (第2次)
    新发现: abc_conf 10维零值
    修复: 12维全真信号

  第5轮: 5对也炸了
    核心问题: 10M完美→40M崩溃, 容量/数据比?
    结论: 不仅容量, 控制bug是主因

  第6轮: 40M容量辩论
    争议: 太大还是太小?
    结论: 双下降"上升期", 应扩到100M+

  第7轮: 梯度诊断结果分析
    数据: bias=2.0梯度死亡 vs bias=0梯度复活
    共识: sigmoid初始化是根因

  第8轮: 容量计算
    DeepSeek: 双下降理论
    千问: 梯度/参数比精确计算分层LR
    实施: P7×0.05, Gate×5.0, ABC×3.0

  第9轮: 收敛共识
    6个P0修复清单确认
    eval脚本维度不匹配真相大白
    CONVERGE

四、辩论直接产出的关键修复

  ✓ gate_base零向量 → abc_confidence 12维真信号
  ✓ 3处gate推理不一致 → 统一384D拼接
  ✓ sigmoid bias=2.0 → bias=0 (梯度复活)
  ✓ LayerNorm除零NaN → 改tanh→最终384D拼接
  ✓ ABC初始化尺度失衡 → abc 0.1→0.01
  ✓ SelfLR非对称 → 对称+上限
  ✓ aux_scale最低值 → 0.02→0.2
  ✓ eval脚本维度不匹配 → 单修复即从2.1%→78.6%
  ✓ 分层LR精确计算 → P7×0.05, 其他×0.5-5.0
  ✓ RMSNorm Q/K防爆 → 千问建议+用户理解实施

五、为什么"前端不丰富但作用极大"?

  1. 快速搭建 (~10分钟)
     400行Python, 无框架依赖, 直接跑

  2. 审计机制防止单AI盲区
     DeepSeek倾向于过度自信, 千问擅长挑错
     联合输出比任一单独准确

  3. 可调用工具验证代码
     千问直接读train_v19_turbo.py验证DeepSeek的计算
     避免了多次"基于错误前提的分析"

  4. 效率
     9轮辩论产出~15个有效修复
     无辩论时同样的问题需要反复手动debug

  5. 记录留存
     每次辩论结果存入logs/
     可回溯每个决策的推理过程

  局限性 (已知):
     - 千问API偶尔拥堵超时
     - DeepSeek有时不读正确代码文件
     - 前端简陋(终端交互)
     - 无多轮对话可视化

  但这些都不影响核心功能——两个AI互相审计, 效果远大于单个。
